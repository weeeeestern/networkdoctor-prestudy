## 리눅스 네트워크 전체 흐름

// 패킷이 들어와서 애플리케이션까지 가는 기본적인 흐름

**NIC → IRQ → softirq(NAPI) → 커널 네트워크 스택 → 소켓 → 애플리케이션**

### NIC

네트워크 카드. 랜카드라고 부르는 것!! 랜선이 꽂히는 포트의 부품이다.

컴퓨터가 인터넷과 연결되어 데이터를 주고받을 수 있게 하는 하드웨어.

외부에서 들어온 패킷을 가장 먼저 받는 하드웨어.

<NIC 단계에서 생기는 문제>

- NIC가 패킷을 버리게 되거나(하드웨어 드롭)
- 받은 패킷을 잠시 쌓아두는 링 버퍼, 링버퍼에 쌓이는 패킷 속도가 CPU/운영체제가 패킷을 가져가는 속도보다 너무 빠를 때 (링버퍼 오버플로)
- 하드웨어는 정상인데, 이를 제어하는 소프트웨어 (드라이버 : NIC와 운영체제가 소통할 수 있게 해주는 프로그램)에 버그가 있거나 버전이 맞지 않을 때 (드라이버/NIC 설정 문제)

관련해서 rocky Linux에서는 NIC 단계에서 수신 중 버려진 패킷의 개수 지표인 `rx_dropped.nic` 를 확인해볼 수 있다. 해당 지표가 올라가면 NIC 단계에서의 문제들을 의심해보아야 하는 것이다.

### IRQ

Interrupt Request.

NIC(랜카드)에 패킷이 도착하면, NIC는 하드웨어 인터럽트를 발생시켜 CPU에 패킷이 왔으니 가져가라는 신호를 보낸다.

<IRQ 단계에서 생기는 문제>

- 요즘 서버들은 멀티 코어인데, 해당 IRQ가 특정 CPU에만 몰릴 때 (IRQ Imbalance)
    - 전체 CPU 사용량을 보면 널널하지만, 네트워크 처리를 전담하게된 특정 코어는 사용률이 어엄~청 높아지니까 지연이 발생함

관련해서 irqbalance 같은 데몬을 사용하거나, 코어별로 인터럽트를 수동 분산 시킬 수 있다고 한다.

### softing/NAPI

소프트웨어적 대용량 처리.

패킷이 막 1초에 10만개씩 들어오는데 모든 패킷마다 NIC가 CPU에 IRQ를 보내면 힘들어지겠지.. 인터럽트는 그리고 무거운 작업이기도 하다.

**NAPI (New API)**

- 패킷이 처음 도착하면 IRQ를 딱 한 번만 보낸다 (Hard IRQ)
- CPU는 이제 하드웨어 인터럽트를 보내지 말라고 신호를 보내고, 링버퍼에 쌓인 수많은 패킷들을 가져오기 시작
- 이렇게 하드웨어의 개입없이, 커널이 주기적으로 링버퍼를 돌면서 패밋을 뭉텅이로 가져온다. 해당 소프트웨어 작업을 soft irq라고 부른다!

커널은 softirq 에도 예산(Budget)을 할당해서 패킷처리에 너무 과부화되지 않도록 조정한다.

관련해서 확인해야하는 리눅스 커널 파라미터들

- `net.core.netdev_budget`(한 번의 softirq 사이클에서 처리할 수 있는 최대 패킷 수)
- `net.core.netdev_budget_usecs`(최대 시간)
- `net.core.netdev_max_backlog`(백로그 최대크기)

※ 링버퍼 : NIC의 버퍼, 백로그 : CPU의 버퍼

### 커널 네트워크 스택

리눅스가 효율적인 이유 중 하나!!

패킷이 커널 네트워크 스택을 지나면서 디캡슐화를 할 때, 바이트 자체로 이동하지 않는다.

리눅스 내부에서는 **skb**(`sk_buff`), socket buffer 라는 자료구조로 표현된다.

실제 데이터는 메모리 한 곳에 가만히 저장시켜두고, 해당 메모리 주소를 가리키는 포인터만 담은 구조체이다.

디캡슐레이션, L2 → L3 → L4 … 계층으로 넘어갈 때마다 실제 패킷 데이터의 바이트를 복사하는 것이 아니라 socket buffer만 복사하고 조작하면서 네트워크 스택 전체를 이동한다!

따라서 패킷이 어디서 드롭되었는가를 추적할 때는 결국 **skb의 생애주기**를 따라가야 한다.

### 소켓

TCP/UDP OSI L4 전송계층의 소켓 단계에서는

- RTT가 얼마나 늘었나
- 재전송(드랍돼서 다시 보내는것)이 있는지
- congestion window (한 번에 보낼 수 있는 패킷 양, TCP는 혼잡을 감지하면 cwnd 크기를 확 줄인다)이 줄었나
- RTO(재전송 Timeout)이 커졌나

와 같은 TCP 연결의 내부 상태 체크가 중요하다.

리눅스에서는 `ss -i` 명령어로 RTT, cwnd 크기 등을 한 눈에 볼 수 있다.

### 어플리케이션

SpringBoot나 Nginx 같은 어플리케이션 단에서는 Timeout(아예 안옴), Latency(지연), Error 등으로만 문제를 인식할 수 있다.

이런 증상들로는 원인을 찾을 수 없고 문제를 해결할 수 없다. 어플리케이션 코드 때문인지, TCP 재전송 때문인지, 커널 드롭 때문인지… 전혀 구분할 수 없다.

그래서 우리같은 운영자들은 NIC부터 어플리케이션까지 올라오는 경로 중 어떤 계층에서 병목이 생겼는지 확인해야 하는 것이다!!
